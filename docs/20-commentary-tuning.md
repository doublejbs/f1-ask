# 20. 해설 프롬프트 개선 — 오프라인 튜닝 하네스

## 배경

벨기에 GP를 프로덕션에 리플레이해 실제 Gemini 해설 29건을 생성했다(변형 `ko:standard`).
목표(환각·헤지 제거, 순위 맥락, 고정 비용)는 검증됐다. 그러나 실데이터에서만
드러난 문제 셋이 있다.

### 1. investigation 해설이 중복된다

조사는 "접수 → 종료" 두 이벤트로 오는데 해설이 거의 같다.

```
 6. HAM과 RUS의 충돌 조사로 페라리의 HAM은 4위 수성에 비상이 걸렸다.
 8. HAM과 RUS의 충돌 조사로 페라리의 HAM은 4위 수성에 비상이 걸렸다.   ← 완전히 동일
21. 4위 HAM의 언세이프 릴리즈 조사는 포디움 추격에 치명적인 변수다.
22. 4위 HAM의 언세이프 릴리즈 조사는 포디움 추격에 치명적인 변수다.   ← 완전히 동일
```

러닝 컨텍스트가 반복을 줄여야 하는데, 두 이벤트가 3분 넘게 떨어져 있어 "최근 4건"
창을 벗어났다. 접수와 종료는 다른 정보인데(종료는 결과가 나왔다), 같은 문장이 된다.

### 2. strategy_note가 과다하고 단조롭다

29건 중 **12건**이 "누구는 남들과 다른 타이어로 반등을 노린다" 패턴의 변주다.

```
BEA는 대다수가 미디엄인 상황에서 하드 타이어로 다른 전략을 구사한다.
PER는 대다수가 미디엄인 상황에서 소프트 타이어로 초반 순위 상승을 노린다.
BOT는 하드 타이어로 다른 전략을 구사한다. …
```

방송도 이 정도 타이어 선택은 보여준다. 해설 대상으로 적절한지 재고가 필요하다.

### 3. 이벤트-스냅샷 시점 어긋남 (근본 문제)

리플레이·워커 모두 **한 폴링 창의 모든 이벤트에 창 끝의 스냅샷 하나**를 공유한다
(`ReplayCommentaryHarness.ts` `options.snapshot`, `CommentaryGeneration.ts` 단일
`snapshot` 인자). 리플레이는 창이 90분 통짜라 오프닝 랩 리타이어 해설에 **최종 순위**가
붙었다("선두 ANT"). 실제 워커는 창이 1분이라 시차가 작지만 원리는 같다.

프롬프트를 아무리 고쳐도 순위 맥락의 시점이 틀리면 소용없다. **튜닝의 전제로 이벤트별
시점 스냅샷이 필요하다.**

## 왜 오프라인 하네스인가

프롬프트를 바꿔가며 수십 번 재생성해 비교해야 한다. 매번 90분 리플레이를 돌리거나
프로덕션에 쓰는 것은 느리고 위험하다.

조사 결과(`buildEvents`·`normalizeOpenF1SnapshotAt`가 순수 함수, 픽스처로 충분): 기존
`BelgianGp2026.json` 만으로 이벤트와 시점 스냅샷을 모두 재구성할 수 있다. **네트워크도
프로덕션도 필요 없다.** Gemini 키만 있으면 몇 초 만에 재생성된다.

## 설계

### 시점 스냅샷

이벤트마다 `normalizeOpenF1SnapshotAt(index, Date.parse(event.timestamp), v)` 를 호출해
**그 이벤트가 발생한 순간의 스냅샷**을 만든다. 단일 최종 스냅샷 공유를 버린다.

이건 하네스 전용이다. **워커 본체(`generateCommentaryForEvents`)는 이번에 고치지
않는다** — 창이 1분이라 급하지 않고, 범위가 커진다. 별건으로 둔다(아래 범위 밖).

### 픽스처 보강

`buildEvents`가 픽스처에서 재생성하지 못하는 해설 대상이 있다.

- **Retirement** — 픽스처에 `sessionResults`(session_result 엔드포인트)가 없어 빠진다.
  리타이어는 해설 가치가 높으므로(실측 3건) 픽스처에 추가한다
- Overtake·TeamRadio는 해설 대상이 아니므로 무관하다

### 오프라인 하네스

env 게이트로 켠다(기존 하네스 방식). 기본 꺼짐.

- 픽스처 → `buildEvents` → 해설 대상 선별(`selectCommentaryEvents`) → 이벤트별 시점
  스냅샷 → `buildCommentaryContext` → Gemini 생성
- **before/after 비교 출력** — 프롬프트를 바꾸면 무엇이 달라졌는지 나란히
- 호출 수 상한(비용 가드), 실패·mock 표시
- 러닝 컨텍스트를 하네스 안에서 시뮬레이트해 반복 제거 효과를 관찰

### 개선 반복

하네스가 서면 프롬프트를 바꿔가며 돌린다. 순서:

1. **investigation 중복** — 접수/종료를 구분한다. 방법 후보: (a) 종료만 해설 (b) 접수는
   "심의 착수", 종료는 "결과" 로 다른 프롬프트 (c) 러닝 컨텍스트에 같은 사건 이력을
   더 오래 남긴다. 실측으로 고른다
2. **strategy_note** — 해설 대상에서 뺄지, 프롬프트로 다양화할지. 방송 중복도가 근거
3. 시점 스냅샷으로 순위 맥락이 정확해졌는지 확인(오프닝 랩 리타이어가 "선두 ANT"가
   아니라 실제 그 시점 순위를 반영하는지)

## 범위 밖

- **워커 본체의 이벤트별 스냅샷** — 별건. 창이 1분이라 시차가 작다. 하네스로 개선한
  프롬프트를 워커에 반영할 때 함께 검토
- 다른 경기 픽스처 — 벨기에 하나로 시작. 과적합이 보이면 늘린다
- `ja`·`en` 변형, 설명수준 변형 — `ko:standard` 로 튜닝 후 확대
- 프로덕션 리플레이 데이터 정리 — 튜닝이 끝난 뒤. 그때까지 배포 앱에 벨기에 GP가
  라이브로 뜬다

## 수용 기준

1. 오프라인 하네스가 네트워크·프로덕션 없이 픽스처만으로 해설을 생성한다.
2. 각 이벤트가 **그 시점의 스냅샷**으로 순위 맥락을 받는다(단일 최종 스냅샷 아님).
3. 리타이어가 픽스처에서 재생성돼 해설 대상에 포함된다.
4. before/after 비교가 출력돼 프롬프트 변경 효과를 볼 수 있다.
5. 호출 수 상한으로 비용이 제한된다. 기본 꺼짐.
6. investigation 중복이 개선된다(측정으로 확인).
7. strategy_note 처리가 정해진다(측정으로 확인).
8. 프롬프트 문구 변경이 회귀 테스트를 깨지 않는다 — 기존 프롬프트 분리·범위 분기 유지.
