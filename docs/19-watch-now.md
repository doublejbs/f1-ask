# 19. 지금 볼 것 — 방송 보완으로 축 전환

## 배경

지금까지 AI 해설은 **이벤트를 설명**했다. 그런데 이 앱은 방송을 보면서 쓰는 세컨드
스크린이다. 해설은 방송에 이미 나온다. 실측에서 나온 문장이 그대로 문제를 보여준다.

> 41랩에 세이프티 카가 발동되며 트랙 상황이 급변합니다.

방송은 이걸 화면과 함께, 더 빠르게, 더 감정적으로 말한다. 텍스트로 뒤늦게 반복하는
것은 경쟁이 되지 않는다.

### 방송의 구조적 한계 — 화면이 하나다

중계 감독이 선두 배틀을 잡으면 P12 는 존재하지 않는 것이 된다. 이것은 실수가 아니라
매체의 한계다.

**실측으로 확인했다.** 2026 벨기에 GP 인터벌 23,116건을 분 단위로 집계한 결과:

```
1초 이내 동시 배틀:  평균 5.6개
                     95% 의 시간에 2개 이상
                     중반 구간은 일관되게 4~6개
```

방송은 그중 **하나**를 보여준다. 평균 4.6개가 매 순간 보이지 않는 채로 진행된다.
여기가 앱이 설 자리다.

## 원칙: 결정론적 코어, 확률적 엣지

이 레포가 이미 내세우는 원칙(docs/02-architecture.md §3.1)을 이 기능에 그대로 적용한다.

"NOR 타이어 28랩째, 뒤차 3.2초, 피트 창 열림" 은 **템플릿으로 나온다.** 스냅샷에
`tireAgeLaps` · `intervalToBehindSeconds` · `pitStopCount` 가 모두 있다. LLM 을 쓰면
느리고, 비용이 들고, 무엇보다 **없는 숫자를 지어낸다** — 실측에서 params 에 없는
"황색기" 를 생성한 전례가 있다(docs/18).

| 층 | 방식 | 담당 |
|---|---|---|
| 감지 · 랭킹 | **결정론. LLM 없음** | 타이어 임계 · 간격 수렴 · 언더컷 위협 · 순위 급변 |
| 트레이드오프 설명 | LLM 사전 생성(전원 대상, 공유) | "지금 들어가면 4위, 안 들어가면 타이어 28랩" |
| "왜?" | LLM 요청 시 | 사용자가 물을 때만 |

**AI 가 제품에서 차지하는 비중이 줄어드는데, 그것이 이 설계의 목적이다.** 더 싸고
빠르고 환각이 없다.

## 설계

### 토대는 개인화가 아니다

**응원하는 드라이버가 없는 사용자가 있다.** 개인화를 토대로 삼으면 처음 켠 사람에게
빈 화면이 된다. 순서를 뒤집는다.

```
토대   필드 전체 주목도 랭킹        ← 즐겨찾기 없어도 동작한다
 +     즐겨찾기 가산점              ← 있으면 내 드라이버가 위로
 +     방송 추정 감점               ← 선두권은 방송이 이미 보여줄 것이다
```

생성은 어차피 20명 전원 대상이므로 **비용과 구현은 개인화 여부와 무관하다.** 즐겨찾기는
랭킹 가중치일 뿐이며, 필터링은 클라이언트에서 공짜로 일어난다. 사전 생성 캐시가 깨지지
않는다.

**방송 추정은 가정이다.** 방송 화면을 알 방법이 없으므로 "감독은 보통 선두를 잡는다" 는
휴리스틱으로 선두권을 감점할 뿐이다. 틀릴 수 있고, 틀리면 이미 본 것을 또 보여준다.
가정임을 코드 주석에도 남긴다.

### 감지기와 임계값 — 실측으로 정한다

벨기에 GP 전체를 돌려 임계값별 발화 수를 측정했다. 아래는 **즐겨찾기 2명 기준
사용자 체감 건수**다.

| 감지기 | 느슨 | 채택 | 빡빡 | 방송이 보여주나 |
|---|---|---|---|---|
| A 타이어 노후 | 15랩 2.8 | **20랩 1.6** | 25랩 1.0 | 거의 안 함(중위권은 전혀) |
| B 간격 수렴 | 0.5초 11.3 | **1.0초 9.5** | 1.5초 8.0 | 자기가 고른 배틀만 |
| C 언더컷 위협 | 3계단 4.5 | **2계단 2.9** | 1계단 1.3 | 거의 안 함 |
| D 순위 급변 | 2계단 11.0 | **3계단 6.2** | 4계단 3.5 | 부분적으로 |

A 와 C 가 가장 값지다. 건수가 적고(합쳐 레이스당 4.5건) 방송이 구조적으로 안 보여준다.
B 와 D 는 건수가 많고 방송과 겹친다.

**임계값은 설정으로 뺀다.** 위 숫자가 실제로 어떤 체감인지는 레이스를 보면서만 알 수
있다. 코드 수정 없이 조절 가능해야 한다.

### 세이프티카 중 배틀 감지를 억제한다

동시 배틀 분포의 꼬리에 21개 · 19개 · 18개가 찍힌 구간이 있다. SC · VSC 로 전 차량이
밀착한 구간으로 보인다. 이때 간격 기반 감지는 무의미하다.

억제하지 않으면 **가장 시끄러운 순간에 가장 쓸모없는 알림이 쏟아진다.**

### 화면: "지금 볼 것" 2~3개 + 행 표시

동시 배틀이 평균 5.6개이므로 감지된 것을 모두 올리면 소음이다.

- **"지금 볼 것" 리스트 — 랭킹 상위 2~3개.** 즐겨찾기가 없어도 항상 채워진다
- **나머지는 순위표 행에 조용히 표시.** 보고 싶은 사람만 본다. 기존 배틀 인라인 표시를
  재사용한다

### 주목도 점수

결정론으로 계산한다. 구성 요소:

- 감지기별 기본 점수 (A · C 가 B · D 보다 높다 — 방송 보완도가 근거)
- 순위 스테이크 (포디움 · 포인트권 경계가 중위권보다 높다)
- 희소성 (드물게 일어나는 상황일수록 높다)
- 즐겨찾기 가산점
- 선두권 감점 (방송 추정)

가중치는 상수로 모아 두고 조절 가능하게 한다.

## docs/18 에서 이어지는 것과 폐기하는 것

**이어진다** — 워커 사전 생성, 러닝 컨텍스트, `aiCommentary` 저장, 호출 타임아웃,
재시도 상한, 변형 설정, 비용 구조. 무엇을 생성하든 필요한 인프라다.

**폐기한다** — **Session 범위 이벤트의 해설.** SC · VSC · 재개 · 플래그는 방송이 가장
잘하는 영역이다. 실측에서 나온 무가치한 문장이 전부 여기였다. `RaceEventScope` 로 이미
갈라 두었으므로 대상에서 빼면 된다.

**관점이 바뀐다** — Driver 범위 이벤트 해설은 남기되 "무슨 일이 일어났는가" 가 아니라
"이 드라이버에게 무슨 의미인가" 로 쓴다.

## 범위 밖

- 방송 화면 인식(불가능하다. 선두권 감점은 휴리스틱일 뿐이다)
- 시청 이력 기반 암묵적 관심 추론
- 자리 비운 동안의 요약(별도 기능으로 검토)
- 알림 푸시(앱 내 표시만)
- 타이어 성능 모델링 · 랩타임 예측

## 알려진 데이터 문제 (별건)

docs/18 에서 이어진다. 이 작업에서 고치지 않는다.

1. `safety_car` 의 `params` 가 `{}` 다 — SC 억제 구현 시 다시 걸릴 수 있다
2. RUS 의 `retirement` 이 `lapsCompleted: 0` 이다

## 수용 기준

1. 즐겨찾기가 없어도 "지금 볼 것" 이 2~3개 채워진다.
2. 즐겨찾기가 있으면 해당 드라이버 상황이 랭킹에서 우선한다.
3. 감지와 랭킹에 LLM 을 쓰지 않는다.
4. SC · VSC 중 간격 기반 감지가 억제된다.
5. 임계값과 랭킹 가중치가 설정으로 조절된다.
6. Session 범위 이벤트의 해설이 생성되지 않는다.
7. 감지된 것 전부가 "지금 볼 것" 에 올라가지 않는다 — 나머지는 행 표시다.
8. 도메인 단위 테스트로 고정: 감지기별 임계 동작 · 랭킹 순서 · SC 억제 · 즐겨찾기 가산점.
9. 실제 레이스 데이터(벨기에 GP)로 감지 건수가 측정치와 일치함을 회귀 테스트로 고정한다.
