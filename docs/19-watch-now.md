# 19. 지금 볼 것 — 방송 보완으로 축 전환

## 배경

지금까지 AI 해설은 **이벤트를 설명**했다. 그런데 이 앱은 방송을 보면서 쓰는 세컨드
스크린이다. 해설은 방송에 이미 나온다. 실측에서 나온 문장이 그대로 문제를 보여준다.

> 41랩에 세이프티 카가 발동되며 트랙 상황이 급변합니다.

방송은 이걸 화면과 함께, 더 빠르게, 더 감정적으로 말한다. 텍스트로 뒤늦게 반복하는
것은 경쟁이 되지 않는다.

### 방송의 구조적 한계 — 화면이 하나다

중계 감독이 선두 배틀을 잡으면 P12 는 존재하지 않는 것이 된다. 이것은 실수가 아니라
매체의 한계다.

**실측으로 확인했다.** 2026 벨기에 GP 인터벌 23,116건을 분 단위로 집계한 결과:

```
1초 이내 동시 배틀:  평균 5.6개
                     95% 의 시간에 2개 이상
                     중반 구간은 일관되게 4~6개
```

방송은 그중 **하나**를 보여준다. 평균 4.6개가 매 순간 보이지 않는 채로 진행된다.
여기가 앱이 설 자리다.

## 원칙: 결정론적 코어, 확률적 엣지

이 레포가 이미 내세우는 원칙(docs/02-architecture.md §3.1)을 이 기능에 그대로 적용한다.

"NOR 타이어 28랩째, 뒤차 3.2초, 피트 창 열림" 은 **템플릿으로 나온다.** 스냅샷에
`tireAgeLaps` · `intervalToBehindSeconds` · `pitStopCount` 가 모두 있다. LLM 을 쓰면
느리고, 비용이 들고, 무엇보다 **없는 숫자를 지어낸다** — 실측에서 params 에 없는
"황색기" 를 생성한 전례가 있다(docs/18).

| 층 | 방식 | 담당 |
|---|---|---|
| 감지 · 랭킹 | **결정론. LLM 없음** | 타이어 임계 · 간격 수렴 · 언더컷 위협 · 순위 급변 |
| 트레이드오프 설명 | LLM 사전 생성(전원 대상, 공유) | "지금 들어가면 4위, 안 들어가면 타이어 28랩" |
| "왜?" | LLM 요청 시 | 사용자가 물을 때만 |

**AI 가 제품에서 차지하는 비중이 줄어드는데, 그것이 이 설계의 목적이다.** 더 싸고
빠르고 환각이 없다.

## 설계

### 토대는 개인화가 아니다

**응원하는 드라이버가 없는 사용자가 있다.** 개인화를 토대로 삼으면 처음 켠 사람에게
빈 화면이 된다. 순서를 뒤집는다.

```
토대   역할이 고정된 칸 (선두권 · 필드)   ← 즐겨찾기가 없어도 채워진다
 +     내 드라이버 칸                     ← 즐겨찾기가 있으면 열리고, 없으면 접힌다
```

**즐겨찾기는 칸 하나의 조건일 뿐 다른 칸에 개입하지 않는다.** 순위를 흔드는 가중치가
아니다. 구체적인 칸 구조와 칸 안 선택 규칙은 §화면 · §칸 안에서 무엇을 고를 것인가 에서
정한다 — 그쪽이 이 문서의 최종 설계다.

생성은 어차피 20명 전원 대상이므로 **비용과 구현은 개인화 여부와 무관하다.** 필터링은
클라이언트에서 공짜로 일어나고, 사전 생성 캐시가 깨지지 않는다.

### 감지기와 임계값 — 실측으로 정한다

벨기에 GP 전체를 돌려 임계값별 발화 수를 측정했다. 아래는 **즐겨찾기 2명 기준
사용자 체감 건수**다.

| 감지기 | 느슨 | 채택 | 빡빡 | 방송이 보여주나 |
|---|---|---|---|---|
| A 타이어 노후 | 15랩 2.8 | **20랩 1.6** | 25랩 1.0 | 거의 안 함(중위권은 전혀) |
| B 간격 수렴 | 0.5초 11.3 | **1.0초 9.5** | 1.5초 8.0 | 자기가 고른 배틀만 |
| C 언더컷 위협 | 3계단 4.5 | **2계단 2.9** | 1계단 1.3 | 거의 안 함 |
| D 순위 급변 | 2계단 11.0 | **3계단 6.2** | 4계단 3.5 | 부분적으로 |

A 와 C 가 가장 값지다. 건수가 적고(합쳐 레이스당 4.5건) 방송이 구조적으로 안 보여준다.
B 와 D 는 건수가 많고 방송과 겹친다.

**임계값은 설정으로 뺀다.** 위 숫자가 실제로 어떤 체감인지는 레이스를 보면서만 알 수
있다. 코드 수정 없이 조절 가능해야 한다.

조절 경로는 `NEXT_PUBLIC_WATCH_NOW_*` 환경변수다(`packages/schemas/src/EnvSchema.ts`).
미설정이면 위 표의 "채택" 열이 그대로 쓰이므로 기본 동작은 바뀌지 않고, 덮어쓸 자리만
열린다. 임계값은 비밀이 아니라 이 문서에 공개된 숫자이므로 클라이언트 번들에 인라인되는
`NEXT_PUBLIC_` 접두사를 써도 잃을 것이 없다.

### 세이프티카 중 배틀 감지를 억제한다

동시 배틀 분포의 꼬리에 21개 · 19개 · 18개가 찍힌 구간이 있다. SC · VSC 로 전 차량이
밀착한 구간으로 보인다. 이때 간격 기반 감지는 무의미하다.

억제하지 않으면 **가장 시끄러운 순간에 가장 쓸모없는 알림이 쏟아진다.**

### 화면: 역할이 고정된 칸 3개 + 행 표시

동시 배틀이 평균 5.6개이므로 감지된 것을 모두 올리면 소음이다. 칸은 3개다.

**칸을 경쟁시키지 않는다.** 하나의 리스트에 모든 후보를 넣고 순위를 매기면 "P16 언더컷과
선두 교체 중 무엇이 위인가" 를 답해야 하고, 그 답은 데이터에 없다. 실제로 초기 구현에서
가중치를 정하려다 근거 없는 숫자(기본 50/30, 스테이크 18/15/8, 선두권 감점 −10,
즐겨찾기 +25)를 만들어 냈다. 전부 추정이었다.

칸에 역할을 고정하면 그 비교 자체가 사라진다.

| 칸 | 범위 | 근거 |
|---|---|---|
| 1 선두권 | P1~P3 | 포디움은 F1 규정이 정한 경계다. 지어낸 값이 아니다 |
| 2 필드 | P4 이하 전체 | 방송이 안 보여주는 나머지. 이 앱의 존재 이유다 |
| 3 내 드라이버 | 즐겨찾기 | 없으면 **칸을 접는다** |

2 칸을 "중위권" 으로 좁히지 않는다. 앱의 근거가 "P12 가 화면에 안 나온다" 인데 중위권만
잘라내면 P15~P20 이 다시 사라진다.

3 칸은 즐겨찾기가 없으면 아무것도 보여주지 않고 접는다. 억지로 채우면 "내 드라이버" 라는
칸의 의미가 무너진다.

**나머지 감지 결과는 순위표 행에 조용히 표시**한다. 보고 싶은 사람만 본다. 기존 행 마커
슬롯(`DriverRowMarkerView`)을 재사용하되, 페널티 · 조사 · 순간 이벤트와 자리를 다투지
않도록 슬롯 위에 얹는 별도 층으로 그린다 — 겨루게 하면 흔한 마커가 있는 행에서 신호가
조용히 사라져 위 약속이 다시 깨진다.

표시는 점 하나이고 주체 드라이버의 행에만 붙인다. 상대역 행에는 붙이지 않는다 —
상대는 요약 문장이 이미 코드로 담고 있고, 양쪽에 붙이면 배틀 한 건이 점 두 개가 되어
"조용히" 가 성립하지 않는다.

### 칸 안에서 무엇을 고를 것인가

칸이 역할을 나눠 가지므로 **칸 안에서만** 고르면 된다. 감지기 종류끼리 점수를 매길
필요가 없다.

칸 안의 선택 규칙은 둘뿐이다. 둘 다 지어낸 값이 아니다.

- **걸린 챔피언십 포인트** — F1 규정이 25/18/15/12/10/8/6/4/2/1 로 정해 놨다.
  P1↔P2 다툼은 7점이, P10↔P11 은 1점이 걸린다. 취향이 아니라 규칙이다
- **최신순** — 포인트가 같으면 방금 일어난 것이 먼저다

**선두권 감점을 폐기한다.** "방송이 선두를 보여줄 것" 이라는 추정이었는데, 칸을 나누면
선두권이 한 칸만 쓰므로 다른 칸을 밀어낼 수 없다. 가정을 코드에 심을 이유가 사라졌다.

**즐겨찾기 가산점도 폐기한다.** 전용 칸이 있으므로 다른 칸의 순서를 흔들 필요가 없다.

다양성 캡은 남긴다. 같은 드라이버가 두 칸에 뜨는 것은 취향 문제가 아니라 화면 낭비다.
**상대역까지 포함해 센다** — 초기 구현은 주체 드라이버만 세어, 한 드라이버가 1 칸에서
상대역 · 2 칸에서 주체로 두 번 뜨는 것을 막지 못했다.

## docs/18 에서 이어지는 것과 폐기하는 것

**이어진다** — 워커 사전 생성, 러닝 컨텍스트, `aiCommentary` 저장, 호출 타임아웃,
재시도 상한, 변형 설정, 비용 구조. 무엇을 생성하든 필요한 인프라다.

**폐기한다** — **Session 범위 이벤트의 해설.** SC · VSC · 재개 · 플래그는 방송이 가장
잘하는 영역이다. 실측에서 나온 무가치한 문장이 전부 여기였다. `RaceEventScope` 로 이미
갈라 두었으므로 대상에서 빼면 된다.

**관점이 바뀐다** — Driver 범위 이벤트 해설은 남기되 "무슨 일이 일어났는가" 가 아니라
"이 드라이버에게 무슨 의미인가" 로 쓴다.

## 범위 밖

- 방송 화면 인식(불가능하다. 무엇이 지금 중계에 나가고 있는지 앱은 알 방법이 없고,
  추정해서 순위를 깎는 방식은 §칸 안에서 무엇을 고를 것인가 에서 폐기했다)
- 시청 이력 기반 암묵적 관심 추론
- 자리 비운 동안의 요약(별도 기능으로 검토)
- 알림 푸시(앱 내 표시만)
- 타이어 성능 모델링 · 랩타임 예측

## 알려진 데이터 문제 (별건)

docs/18 에서 이어진다. 이 작업에서 고치지 않는다.

1. `safety_car` 의 `params` 가 `{}` 다 — SC 억제 구현 시 다시 걸릴 수 있다
2. RUS 의 `retirement` 이 `lapsCompleted: 0` 이다

## 수용 기준

1. 즐겨찾기가 없어도 선두권 · 필드 칸이 채워진다.
2. 즐겨찾기가 없으면 세 번째 칸이 접힌다. 억지로 채우지 않는다.
3. 감지와 랭킹에 LLM 을 쓰지 않는다.
4. SC · VSC 중 간격 기반 감지가 억제된다.
5. 감지 임계값이 코드 수정 없이 설정으로 조절된다. (랭킹 가중치는 폐기됐으므로 없다 —
   §칸 안에서 무엇을 고를 것인가)
6. Session 범위 이벤트의 해설이 생성되지 않는다.
7. 감지된 것 전부가 칸에 올라가지 않는다 — 나머지는 **실제로 순위표 행 표시로 간다.**
   계산만 하고 아무도 읽지 않으면 이 기준을 만족하지 않는다. 칸당 줄 수를 좁게 잡은
   근거가 "나머지는 행에서 볼 수 있다" 이므로 이 절반이 없으면 정보가 사라진다.
8. 칸에 이미 뜬 신호는 행에 중복 표시되지 않는다.
9. 행 표시는 칸보다 시각적으로 가볍고, 순위표의 행 높이와 열 폭을 바꾸지 않는다.
10. 칸 사이에 점수 비교가 없다. 감지기 종류별 가중치 · 선두권 감점 · 즐겨찾기 가산점이
    코드에 없다.
11. 같은 드라이버가 두 칸에 뜨지 않는다. 상대역으로 등장하는 경우도 포함한다.
12. 도메인 단위 테스트로 고정: 감지기별 임계 동작 · 칸 배정 · 칸 안 정렬(포인트 → 최신) ·
    SC 억제 · 즐겨찾기 없을 때 칸 접힘 · 칸과 overflow 가 겹치지 않고 후보를 잃지 않음.
13. 실제 레이스 데이터(벨기에 GP)로 감지 건수가 측정치와 일치함을 회귀 테스트로 고정한다.
