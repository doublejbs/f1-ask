# 16. 폴러 워커 승격 (Cloud Functions)

## 배경

폴러는 지금 `packages/domain/test/OpenF1LivePoll.test.ts` — **vitest 테스트 파일**이다.
파일 주석에 "Worker 역할"이라 써 있지만 실체는 사람이 터미널에서 띄우는 개발 하네스다.
터미널이 꺼지면 데이터가 멈춘다.

또한 실제 Firestore 로 전환하면서 **서비스 계정 키 전달이 막혔다**. Cloud Functions
런타임은 ADC 를 자동으로 받으므로 **키 파일 자체가 필요 없어진다.** 두 문제가 같이 풀린다.

## 배포 제약 — 번들링이 필수다

두 가지가 겹친다.

1. `packages/domain/package.json` 의 `main` 이 `./src/Domain.ts` 다. 빌드 산출물이
   없고 TypeScript 소스를 그대로 내보낸다. Next.js 는 `transpilePackages` 로 처리하지만
   Functions 런타임은 TS 를 실행하지 못한다.
2. Firebase 배포는 npm 으로 설치를 시도하는데 `workspace:*` 프로토콜을 이해하지 못한다
   (호스팅에서 이미 겪었다 — [09-deployment.md](09-deployment.md)).

따라서 함수는 **esbuild 로 단일 JS 파일로 번들**한다. 도메인 소스가 인라인되므로
워크스페이스 의존이 사라지고 TS 도 함께 해결된다.

- 번들 대상: 함수 엔트리 + `@f1/domain`
- external: `firebase-admin`, `firebase-functions` (런타임이 제공)
- `firebase.json` 의 `functions.predeploy` 에서 번들을 실행한다

## 스케줄 설계

Cloud Functions 는 무한 루프를 돌 수 없다. **1분마다 깨어나 6초 간격으로 폴링하고 종료**한다.

```
Cloud Scheduler (1분) → 함수 기동 → 세션 활성 확인
                                  ├ 비활성 → 즉시 종료 (비용 0에 수렴)
                                  └ 활성   → 6초 간격 10회 폴링 후 종료
```

### 비활성 시 즉시 종료 — 비용 가드

F1 세션은 드물다. 레이스가 없는 동안 24시간 폴링하면 OpenF1 API 와 Firestore 를
헛되이 두드린다. **세션이 진행 중이 아니면 아무 것도 하지 않고 끝낸다.**

판정은 `fetchLatestOpenF1Meta` 의 세션 시각을 쓴다. 세션 종료 시각을 지났으면 비활성으로
본다. 판정 여유(예: 종료 후 N 분까지는 활성으로 간주)를 두어 마지막 데이터가 잘리지 않게 한다.

### 중복 실행 방지

스케줄러 재시도나 실행 지연으로 두 인스턴스가 겹칠 수 있다. 이벤트는
`deduplicationKey` 를 문서 id 로 쓰므로 **재기록은 멱등**이지만, 같은 일을 두 번 하는 것은
낭비다. `firestore.rules` 에 이미 서버 전용으로 준비된 `workerLeases/{sessionId}` 를 쓴다.

- 실행 시작 시 리스를 잡고(만료 시각 포함), 이미 유효한 리스가 있으면 종료
- 리스는 함수 최대 실행 시간보다 짧게 잡아 죽은 인스턴스가 영구 점유하지 않게 한다

## 쓰기 증폭 정리 (같이 고친다)

실측에서 **1회 계산은 622건인데 실제 문서는 843건**이 쌓였다. 원인은
`pushLatestPerDriver` 의 "드라이버당 최근 N건" 창이 폴링마다 이동하면서 **매번 다른
`deduplicationKey` 를 만들기** 때문이다. 상한은 1회 호출 안에서만 유효하고 누적은 막지 못한다.

에뮬레이터에서는 무료였지만 이제 실제 과금 대상이다. 고친다.

- `PersonalBestLap` · `TeamRadioPosted` · `GapClosing` · `OverrideRangeEntered` 등
  창 기반 이벤트의 `deduplicationKey` 를 **발생 시점 기준 고정값**으로 만든다.
  같은 사건은 몇 번을 재계산해도 같은 키가 나와야 한다.
- 키가 고정되면 재기록이 덮어쓰기가 되어 문서 수가 1회 계산값으로 수렴한다.
- 도메인 단위 테스트로 고정한다: 같은 입력을 두 시점에 계산해도 키 집합이 같을 것.

## 범위 밖

- **AI 해설의 워커 생성** — 지금은 클라이언트가 각자 LLM 을 호출해 시청자 수에
  비례해 비용이 는다(레이스당 1인 약 $3). 워커에서 한 번 생성해 Firestore 에 저장하면
  고정비가 된다. 효과가 가장 크지만 locale × 설명수준 9개 변형과 캐시 설계가 얽혀
  별도 단계로 뺀다. `aiCache` 컬렉션은 규칙에 이미 준비되어 있다.
- 리플레이 모드의 워커 이식(리플레이는 로컬 개발 도구로 남긴다)

## 수용 기준

1. 함수가 esbuild 번들로 배포되며 `workspace:*` 없이 설치된다.
2. 서비스 계정 키 파일 없이 동작한다(런타임 ADC).
3. 세션이 비활성이면 폴링 없이 즉시 종료한다.
4. 활성이면 스냅샷과 이벤트가 실제 Firestore 에 기록된다.
5. 리스로 중복 실행을 막는다.
6. 창 기반 이벤트의 `deduplicationKey` 가 재계산에 안정적이다 — 도메인 테스트로 고정.
7. 기존 로컬 리플레이 하네스가 계속 동작한다.
