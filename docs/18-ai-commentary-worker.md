# 18. AI 해설 워커 이전 + 맥락 강화

## 배경

해설은 지금 **시청자 브라우저가 요청할 때** 생성된다. `useRaceCommentary` 가 새 이벤트를
감지해 `/api/commentary` 로 POST 하고, 라우트가 LLM 을 호출한다. 캐시가 없어 비용이
**해설 대상 47건 × 시청자 수**로 는다.

비용보다 큰 문제는 **품질의 천장**이다. 실측(Gemini `gemini-3.5-flash`, 2026 벨기에 GP
실제 이벤트)에서 두 가지가 드러났다.

### 1. 헤지 표현 — 프롬프트를 Q&A 와 공유한 탓

`SYSTEM_RULES` 가 `answerQuestion` 과 `generateCommentary` 에 공용이다. 그 안에
"데이터가 부족하면 모른다고 말하라" 가 들어 있다. Q&A 에서는 옳지만 해설에서는
아무도 묻지 않았는데 변명을 하게 만든다.

> 러셀의 리타이어로 상위권 순위 다툼이 급변했으나, 남은 3랩 동안의 구체적인 피트 전략
> 변화는 **현재 데이터로 확인하기 어렵습니다.**

해설 전용 규칙으로 분리하면 사라진다(실측 7/7).

### 2. 맥락 부족 — 단발 호출의 구조적 한계

해설 프롬프트가 받는 것은 **이벤트 타입 + params + 현재 랩**이 전부다. 순위도 갭도
타이어도 없다. 이 입력으로 "전략적 의미"를 요구하니 모델의 선택지는 둘뿐이었다.
헤지하거나, 이벤트를 한국어로 되뇌거나. 후자는 이벤트 피드가 이미 하는 일이라
해설의 존재 이유가 없어진다.

순위표를 주면 실제로 달라진다.

> HAM 의 5초 페널티로 **1.743초 차인 5위 PIA 가 4위 자리를 맹추격**한다.

그러나 드라이버가 없는 세션 이벤트(SC · VSC · 재개)에서는 같은 맥락이 역효과였다.
연속된 4건이 모두 같은 선두 갭을 되풀이했다. **매 호출이 서로를 모르기 때문**이다.

이 두 번째 문제는 프롬프트로 못 고친다. 생성 주체가 경기 흐름을 들고 있어야 한다.
그래서 워커 이전과 맥락 강화는 같은 작업이다.

## 설계

### 생성 주체: 폴러 워커

`functions/` 의 폴러가 해설을 생성한다. 워커는 이벤트를 **시간순으로** 처리하므로
직전 해설이 자연스럽게 손에 있다. 시청자 요청 시점 생성으로는 이 구조를 만들 수 없다 —
누가 언제 접속했는지에 따라 맥락이 달라진다.

비용은 **레이스당 고정**이 된다. 시청자 수와 무관하다.

**생성 시점: 폴링 루프 안이 아니라 루프가 끝난 뒤다.**

폴링 간격은 6초다. 루프 안에서 LLM 을 기다리면 호출 하나(최대 12초)가 그 간격을 통째로
밀어 데이터가 성겨진다. 이벤트 여러 건이 한 폴링에 잡히면 창 하나가 통째로 무너진다.
그래서 한 기동은 이렇게 나뉜다.

```
0초 ─── 폴링 루프 (6초 × 10회) ─── 60초 ─── 해설 일괄 생성 ─── 최대 96초 ─── 마무리 쓰기
```

**대가**: 해설이 이벤트보다 최대 약 70초 늦게 뜬다. 이벤트는 폴링 즉시 노출되고
해설만 뒤따라 붙는다. 이 지연을 줄이겠다고 루프 안으로 생성을 되돌리면 폴링 간격이
깨진다 — 되살리지 말 것. 지연을 줄여야 한다면 폴링과 해설을 **다른 기동으로 분리**하는
방향이 맞다.

### 저장: `sessions/{sessionId}/aiCommentary/{docId}`

`aiCache` 컬렉션이 규칙에 준비되어 있으나 `allow read, write: if false` 라
클라이언트가 읽지 못한다. 해설은 이벤트와 함께 화면에 뜨는 **공개 읽기** 데이터이므로
이벤트와 같은 자리에 둔다.

**새 컬렉션을 만들지 않는다.** `sessions/{sessionId}/aiCommentary` 는 `firestore.rules`
와 docs/02-architecture.md §44 · docs/03-firestore-and-auth.md 에 이미 설계되어 있고
규칙(공개 읽기 · 서버 전용 쓰기)도 이 용도와 정확히 같다. 아직 쓰는 코드가 없었을 뿐이라
워커가 그 자리를 채운다.

- `docId` 는 `{eventId}:{locale}:{explanationLevel}` — 재생성이 멱등이다
- 규칙은 이벤트 컬렉션과 동일하게 공개 읽기 · 서버 전용 쓰기
- 인덱스가 필요하면 `firestore.indexes.json` 에 추가한다

### 변형: 검증 단계는 `ko × standard` 한 조합만

locale 3 × 설명수준 3 = 9 변형이면 레이스당 423 회다. Gemini 무료 티어 한도와
지출 상한($5)을 감안해 **검증 단계는 1 변형(47 회)** 으로 시작한다. 생성 대상 변형은
설정으로 바꿀 수 있어야 한다 — 코드 수정 없이 늘릴 수 있어야 한다.

없는 변형을 클라이언트가 요청하면 해설 없이 이벤트만 보여준다(아래 폴백).

### 맥락 번들

프롬프트에 넣을 것. **실측으로 효과가 확인된 것만 넣는다.**

| 항목 | 출처 | 근거 |
|---|---|---|
| 이벤트 타입 · params | 현행 | — |
| 세션 상태 · 현재/전체 랩 | 현행 | — |
| 관련 순위 슬라이스 | `snapshot.drivers` | Driver 이벤트에서 의미가 생김(실측) |
| 직전 해설 N 건 | 러닝 컨텍스트 | 연속 이벤트 반복 제거 |
| 누적 경기 상태 | 이벤트 누적 | 리타이어 수 등 흐름 |

순위 슬라이스는 상위 3명 + 해당 드라이버 앞뒤로 좁힌다. 전체 20명을 넣으면 토큰만
늘고 모델이 초점을 잃는다.

`LiveDriverState` 의 `compound` · `tireAgeLaps` · `pitStopCount` · `intervalToAheadSeconds`
는 전략적 의미를 크게 키울 후보다. 다만 **아직 실측하지 않았다.** 1차에서는 넣지 않고,
워커 이전 후 실제 세션 데이터로 확인한 뒤 별도로 판단한다.

### 러닝 컨텍스트의 저장

Cloud Functions 는 1분마다 새 프로세스로 뜬다. 직전 해설을 메모리에 둘 수 없다.
`runtime/eventCursor` 와 같은 방식으로 `runtime` 문서에 보관한다.

- 최근 생성한 해설 텍스트 N 건과 누적 카운터를 담는다
- 폴링 창 시작에 한 번 읽고 끝에 한 번 쓴다 — 이벤트마다 쓰면 커서 문서가
  쓰기 폭증이 된다(docs/16-poller-worker.md 에서 같은 실수를 이미 겪었다)
- 이미 저장한 해설 문서 id 와 **실패 횟수**도 같은 문서에 얹는다. 창당 읽기 1 · 쓰기 1
  안에서 끝나므로 중복 방지와 재시도 상한에 추가 I/O 가 한 건도 들지 않는다

**직전 해설은 변형별로 나눠 담는다.** 변형(locale × 설명수준)은 서로 다른 독자를 향한
다른 글이다. 평평한 배열 하나에 섞으면 두 가지가 동시에 깨진다.

1. 한국어 해설이 영어 프롬프트의 "직전 해설" 로 들어간다
2. 상한 N 건을 변형들이 나눠 써 변형당 실효 건수가 줄고, 반복 제거 효과가 조용히 반감된다

기본값(`ko × standard` 한 조합)에서는 드러나지 않는다. 변형을 늘리는 순간 드러나므로
설계 단계에서 갈라 둔다.

### 프롬프트: 범위로 가른다

`RaceEventScope` 가 이미 `Session` / `Driver` 로 존재한다. 새 분류를 만들지 않는다.

- **Driver** — 순위 맥락을 주고 "왜 중요한지"를 쓰게 한다
- **Session** — 맥락 없이 사실만. 순위를 주면 연속 이벤트가 같은 말을 반복한다(실측)

공통 규칙으로 확정된 것(모두 실측 근거가 있다):

- 데이터에 없는 것은 존재하지 않는다 — 깃발 · 날씨 · 타이어를 지어내지 않는다
  (`track_hazard` 에서 params 에 없는 "황색기"가 나왔다)
- 데이터 · 출처 · 불확실성을 언급하지 않는다
- 드라이버는 주어진 3글자 코드로만 지칭한다 (현행은 "러셀(RUS)" · "해밀턴" 혼용)
- 정확히 한 문장, 종결어미 `다` (명사형으로 튀는 사례가 있었다)

`ClaudeProvider` 와 `GeminiProvider` 가 같은 문구를 공유하므로 **함께 고친다.**
한쪽만 바꾸면 provider 를 바꿨을 때 품질이 달라진다.

### 클라이언트: 구독으로 전환

`useRaceCommentary` 가 POST 대신 Firestore 를 구독한다.

**단, mock · replay 모드에는 워커가 없다.** 이 모드에서는 기존 POST 경로를 유지한다.
`DATA_MODE` 로 분기한다. 로컬 개발이 깨지면 안 된다.

### 폴백

생성 실패가 이벤트 표시를 막으면 안 된다.

- LLM 실패 → 해당 이벤트의 해설을 **쓰지 않는다.** 이벤트는 그대로 노출된다
- **mock 텍스트를 Firestore 에 저장하지 않는다.** 저장하면 키를 고친 뒤에도
  "경기의 주목할 만한 순간입니다" 가 영구히 남는다
- 해설 생성이 폴링을 막지 않는다. 함수 잔여 시간을 넘기면 남은 이벤트는
  다음 기동이 커서로 이어받는다

#### LLM 호출에 상한 타임아웃을 건다

"잔여 시간을 넘기면 다음 기동으로 넘긴다" 는 **호출 하나의 최대 소요를 알아야** 성립한다.
상한이 없으면 남은 시간을 검사해도 초과를 막지 못한다 — 검사를 통과한 호출 하나가 함수
타임아웃까지 늘어질 수 있기 때문이다. 그렇게 강제 종료되면 `finally` 가 실행되지 않아
**이벤트 커서 쓰기까지 함께 날아가고**, 다음 기동이 이벤트를 통째로 다시 쓴다
(docs/16 에서 쓰기가 200배 났던 상황과 같다).

- 세 provider(Gemini · Claude · OpenAI) **모두**에 요청 타임아웃을 건다. 한 곳만 걸면
  provider 를 바꿨을 때 다시 깨진다
- 타임아웃 값과 예산 계산의 호출 소요는 **같은 상수**에서 나온다. 두 값이 갈리면
  다음 사람이 한쪽만 고친다
- 응답 헤더뿐 아니라 **본문 읽기까지** 상한 안에 넣는다
- 타임아웃 실패는 다른 실패와 같은 경로를 탄다 — 저장 생략, 폴링 계속

#### 실패한 해설의 재시도 상한

실패한 해설은 러닝 컨텍스트에 남지 않으므로 다음 폴링 창에 다시 잡힌다. 일시적 장애
(쿼터 · 네트워크)에는 이것이 맞지만, **결정론적으로 실패하는 이벤트**(콘텐츠 필터,
이상한 `params` 로 인한 빈 응답)는 레이스가 끝날 때까지 매 창마다 다시 호출된다.
90분 레이스면 이벤트 1건이 최대 90회다. "레이스당 47회 고정" 이 무너지고, 그 호출이
예산을 먹어 신규 이벤트가 굶는다(창당 처리 가능 건수는 약 5건).

- 러닝 컨텍스트에 해설 문서 id 별 **실패 횟수**를 기록한다
- 상한(3회)을 넘기면 그 해설을 **영구히 포기한다.** 이벤트는 해설 없이 노출된다
- 성공하면 그 문서의 실패 이력을 지운다
- 실패 횟수는 이미 창당 1회 쓰는 문서에 얹으므로 **추가 I/O 는 0** 이다
- 실패에는 mock 폴백도 포함한다. 워커는 `FallbackLlmProvider` 를 쓰므로 실제 LLM 오류가
  예외가 아니라 `isMock: true` 로 돌아온다 — 이것을 세지 않으면 상한이 프로덕션에서
  한 번도 발동하지 않는다

## 범위 밖

- 타이어 · 피트 맥락 주입 (실측 후 별도 판단)
- 9 변형 전체 생성 (검증 후 확대)
- 경기 요약(`generateSummary`) 의 워커 이전
- Q&A(`answerQuestion`) 는 성격상 요청 시점 생성이 맞다 — 건드리지 않는다
- 리플레이 하네스의 워커 이식

## 알려진 데이터 문제 (별건)

해설과 독립적이나 해설 품질에 직접 드러난 것들이다. 이 작업에서 고치지 않는다.

1. `safety_car` 의 `params` 가 `{}` 다. 발령 랩도 회차도 없다
2. RUS 의 `retirement` 이 `lapsCompleted: 0` 이다. 41 랩 이벤트와 모순이다

## 수용 기준

1. 워커가 이벤트 기록 시 해설을 생성해 Firestore 에 저장한다.
2. 같은 이벤트를 재처리해도 문서가 늘지 않는다(멱등).
3. 클라이언트가 live 모드에서 구독으로 해설을 받는다. LLM 을 직접 부르지 않는다.
4. mock · replay 모드가 기존 POST 경로로 계속 동작한다.
5. 생성 실패 시 해설이 비고 이벤트는 정상 노출된다. mock 텍스트가 저장되지 않는다.
6. 해설 프롬프트가 Q&A 와 분리된다. Q&A 의 "모르면 모른다" 규칙은 유지된다.
7. 프롬프트가 `RaceEventScope` 로 갈린다.
8. 러닝 컨텍스트가 `runtime` 문서에 보관되고, 폴링 창당 읽기 1 · 쓰기 1 이다.
9. 생성 변형이 설정으로 조절된다. 기본값은 `ko × standard`.
10. 해설 생성이 폴링 루프를 밀지 않는다. 루프가 끝난 뒤 남은 시간 안에서만 돈다.
11. 세 provider 모두 LLM 호출에 상한 타임아웃이 걸린다. 그 값과 워커의 호출 예산이
    같은 상수에서 나온다.
12. 같은 해설이 3회 실패하면 더 이상 호출되지 않는다.
13. 변형이 둘 이상일 때 각 변형이 **자기** 직전 해설 N 건만 받는다. 변형 간에 맥락이
    섞이지 않고 N 건 상한을 나눠 쓰지 않는다.
14. 한 기동의 최대 실행 시간이 워커 리스 TTL 안에 들어간다. 정상 종료는 리스를 즉시 푼다.
15. 도메인 단위 테스트로 고정: 맥락 조립 · 프롬프트 분기 · 멱등 문서 id ·
    호출 타임아웃 · 재시도 상한 · 변형별 맥락 격리.
