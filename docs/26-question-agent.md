# 26. 질문 에이전트 — 툴로 전체 이력을 조회한다 (2단계)

## 배경

docs/25(1단계)로 AI 는 경기 **전체 서사**(선두변경·리타이어·피트웨이브 등 집계)를 안다.
하지만 **임의 깊이 질문**은 집계로 안 된다:

- "VER 몇 랩에 피트했어?" — 드라이버별 개별 피트 랩
- "3위랑 4위 페이스 차이" — 드라이버별 랩타임 이력
- "누가 언더컷 성공했어?" — 스틴트·피트·순위 교차

지금 `answerQuestion` 은 **단발**이다: 스냅샷 + 최근 이벤트 40건을 프롬프트에 넣고 LLM 을
1회 호출한다. 14랩의 피트 이벤트는 후반엔 40건 창 밖으로 밀려 답할 수 없다.

**핵심: 데이터가 없는 게 아니라 전달이 안 되는 것이다.** 드라이버별 이벤트가 이미
Firestore(`sessions/{id}/events`)에 있다. 40건만 보내니 못 볼 뿐이다. 2단계 = **모델이
전체 이력을 조회하게 한다.**

단, 이벤트 종류마다 담는 필드가 다르다 (관찰·타당성 리뷰 실측 — 서두를 단순화하지 말 것):

| 이벤트 | driver | lap | 라이브 존재 | 비고 |
|---|---|---|---|---|
| pit_stop | O | **O** | O | 28건, 전부 랩 정확 — 이 툴의 주력 |
| **investigation** | O | 조건부 | O | **11건, 사유·turn·상대차까지. 인과의 실질 재료(penalty보다 값짐)** |
| penalty | O | 조건부 | O | 1건뿐(사유는 완전) |
| overtake | O | **없음** | O | lapNumber 미기록 → "몇 랩에 추월" 불가 |
| fastest_lap | O | **없음** | O | 랩은 `narrative.fastestLap.lap` 에만 |
| retirement | O | O | **아니오** | sessionResults(종료 후)에서만 — **라이브 events 에 없다.** 라이브 리타이어는 `narrative.retirements`(랩 정체) |
| safety_car·vsc | — | **없음** | O | params `{}` — 랩은 `narrative.safetyCars[].startLap` 에만 |

즉 툴(events 조회)과 프롬프트-실린-데이터(narrative)의 **역할이 타입마다 갈린다.** narrative 가
이미 인과·SC·패스티스트의 랩·순위 이동을 담아 **툴 없이 부분 답이 되는 경우가 많다**(실측:
B 유형 부분율 71%의 상당수가 narrative 덕).

## 방식: 툴 호출 (서버가 Firestore 조회) — 사용자 결정

모델이 필요한 것만 툴로 가져온다. 매 질문에 전체 이력을 다 보내는 것(요청 비대·경기 길수록
증가)보다, "VER 물으면 VER 것만 조회"가 지연·비용에 유리하다. 이 레포의 결정론 원칙과도
맞다 — **툴은 조회·계산된 사실만 반환, LLM 은 추론만.**

## 우선순위 질문 3유형과 데이터 출처 (사용자 선택)

세 유형이 **서로 다른 출처**를 요구한다. 이게 이 스펙의 핵심 구조다.

### A. 전략·비교 — Firestore 이벤트/텔레메트리 (63% 완전·88% 부분, 실측)

"누가 언더컷?", "두 타이어 전략 중 뭐가 빨라?", "3·4위 페이스 차이".

- **이미 답 가능**: "VER 몇 랩 피트"·"가장 오래 탄 타이어"·"HAM 스틴트 전략"·"피트 안 한
  선수"·"누가 소프트로 갈아탐" — pit_stop 이벤트 + **이미 프롬프트에 실린** `contextSummary.
  stints`(22명 전원) + narrative 로 답된다
- **막힌 것은 페이스 비교뿐**: "3·4위 페이스 차이", "미디엄 vs 하드 누가 빨라", "언더컷 델타
  확증" — 드라이버별 **랩타임 이력**이 필요한데 스냅샷엔 최근 3랩(`recentLapTimesSeconds`)뿐

**랩타임 저장 결정 (관찰 실측으로 확정):** 워커는 **이미 매 폴마다 `data.laps` 875행(드라이버당
44랩 전체)을 읽는다** — slice(-3)로 버릴 뿐이다(OpenF1 추가 요청 0, overtake_forecast 도 이미
이 전체 laps 소비). 따라서:

- 전체 랩타임 상시 저장은 **문서 팽창**(22명×44 float) 때문에 유보 — Firestore 1MB·쓰기 비용
- **채택**: (1) 스냅샷에 **스틴트별 페이스 요약**(평균·베스트 랩) 확대 저장 — 팽창 최소로
  페이스 비교 대부분 커버, (2) 세밀 구간 비교는 `queryPace` 툴이 질문 시 **lap 엔드포인트
  1요청**만 온디맨드(docs/22 의 "11요청 금지"를 재도입하지 않는 단일 드라이버·단일 요청 봉투)

### B. 사건 인과 — 29% 완전·71% 부분 (핵심 인과는 0%, 정직한 한계)

"세이프티카 왜 나왔어?", "RUS 리타이어 원인?", "이 페널티 왜?".

- **답 가능**: "HAM 페널티 왜?"(penalty 사유 완전), "LEC/PIA 접촉?"(investigation 완전),
  "NOR 왜 순위 떨어짐?"(narrative.biggestMovers — 실은 상승, 전제 정정)
- **핵심 인과는 0% (실측 재확인)**: "RUS 왜 리타이어"·"SC 왜 나옴" — retirement reason 은
  `dnf`/`dsq` **플래그일 뿐 원인이 아니고**, safety_car params 는 `{}`(원인·랩 없음). OpenF1 이
  안 준다(docs/25 §리타이어와 동일 벽)
- **상관 제시하되 단정 금지 (이게 UX 를 살리는 주 경로)**: 반복적 "기록 없음"은 제품이
  멍청해 보인다. 대신 시각·차량번호로 교차하는 인접 사건을 제시한다 — RUS 리타이어(lap1) ↔
  investigation(turn6 collision), SC(lap1) ↔ track_hazard(recovery_vehicle lap2). **"~때문일 수
  있다"까지, "~때문이다"는 금지.** investigation(11건)이 인과의 실질 재료다(penalty 1건보다 값짐)

### C. 규정·상식 — 큐레이션 지식 (경기 데이터 아님, Firestore 무관)

"오버테이크 모드가 뭐야?", "이 서킷 특징?", "포인트 어떻게 매겨져?".

- 이건 **정적 F1 지식**이라 Firestore·텔레메트리에 없다. docs/22 가 "별도"로 미뤄둔
  큐레이션 지식 파일이 여기서 필요해진다
- **큐레이션 지식 파일**(그린필드): 2026 규정 요지(오버테이크 모드·타이어 규칙·포인트
  25/18/15…·패스티스트랩 포인트 유무)·서킷 성격(레이아웃·추월 지점·과거 특징). 도메인에
  정적 데이터로 두고 `lookupF1Knowledge(topic)` 툴로 조회
- 서킷별은 세션 메타(circuitName, 예: `spa-francorchamps`)로 해당 서킷 항목만 로드
- **신뢰 경계가 다르다 (R5)**: A·B 는 텔레메트리 산수라 결정론이 환각을 막지만, **C 는 편집된
  정적 텍스트라 결정론이 안 지켜준다** — 저자가 틀리면 오류가 "지식"으로 세탁된다. 그래서
  **모든 항목에 출처 주석 필수**(테스트로 강제, 수용 기준 §참조). 검증된 사실만 넣는다

### D. 예측·반사실 — 대부분 overtake_forecast 로 연결 (신규 유형, 실측)

"누가 이길까?", "OO 피트했어야 했나?", "NOR가 VER 잡을까?".

- **결정론 재료가 없다**: 우승 예측·반사실("피트했어야 했나")은 랩타임 델타·시뮬 재료가
  없어 **불가**. 단정하면 환각이다
- **인접 페어 클로징은 기존 것으로**: "NOR가 VER 잡을까"·"지금 페이스면 몇 위"는 정확히
  `overtake_forecast`(docs/23, chaser/target/predictedLap)의 대상이다 — 툴을 새로 만들지 말고
  **기존 예측을 답변에 연결**한다(스냅샷 `overtakeForecasts` 이미 실림)
- 전체 결과 예측은 **범위 밖**으로 명시(정직한 한계)

## 아키텍처

### 툴 호출 추상화 (provider 3곳)

`RaceLlmProvider` 에 툴 루프를 얹는다. 세 provider 의 function-calling wire 포맷이 다르다
(Gemini functionDeclarations · Claude tools/tool_use · OpenAI tools/function). **공용 툴 정의
+ provider별 어댑터**로 나눈다 — 툴 스키마·실행은 한 곳, wire 변환만 provider별.

- 새 메서드보다 `answerQuestion` 내부에 툴 루프를 두는 방향 검토(기존 시그니처 유지).
  루프: LLM 응답에 tool_use 있으면 → 서버가 툴 실행 → 결과를 대화에 추가 → 재호출, 없으면 종료
- **루프 종료 상한 필수 (D1)**: 모델의 자발적 중단만 믿으면 툴→툴→툴 무한 반복으로 비용·지연이
  무계다. **최대 툴 라운드 캡**(예: 3라운드)을 두고, 넘으면 그때까지 모은 tool-result 로
  **강제 최종 답변**한다. 상한 도달·초과 동작을 테스트로 고정(수용 기준 §참조)
- **MockLlmProvider 도 인터페이스를 만족해야** 한다(툴 없이 기존 답). FallbackLlmProvider 통과
- **멀티턴/툴 상태 경계 (D3)**: 각 질문은 독립이다 — 이전 턴의 tool-result 를 이월하지 않고,
  후속 질문은 툴을 재실행한다. 크로스턴 툴 상태 없음(현행 answerQuestion 이 무상태 단일턴인
  것을 유지). 대화 히스토리는 지금처럼 Q&A 텍스트만 이월
- **Gemini generate() 리팩터링 필요 (프로토타입 범위)**: 현재 `GeminiProvider.generate()` 는
  응답 parts 에서 `part.text` 만 뽑고 functionCall part 를 버린다. 툴 루프 최소 구현조차 이
  헬퍼가 functionCall part 를 노출하도록 고쳐야 한다
- **claude-api 스킬 참조**: 구현 시 각 provider tool-calling 정확한 wire 포맷을 스킬/공식 문서로
  확인한다(추정 금지 — 3.x Gemini 사고 토큰 때 데인 교훈)

### 서버측 Firestore (신규 인프라)

지금 `/api/ask` 는 서버 Firestore 접근이 없다(웹은 전부 클라이언트 SDK). 툴이 서버에서
전체 이력을 읽으려면 **firebase-admin 을 Vercel 서버 라우트에 붙인다**:

- 서비스 계정 자격을 Vercel 서버 환경변수로(클라이언트 노출 금지, `NEXT_PUBLIC_` 아님)
- 툴은 `sessions/{sessionId}/events` 를 드라이버·타입·랩 범위로 쿼리. sessionId 는 요청
  스냅샷의 `sessionId` 필드(필수)에서 온다
- **읽기 전용**. 쓰기 없음. driverNumber+type+lapNumber 범위 쿼리는 **복합 인덱스 배포가
  별도 단계**(`firestore.indexes.json`) — 코드 머지≠인덱스 배포(aiCommentary 때 데인 것, 수용
  기준 §참조)
- **sessionId 신뢰 경계**: 현재 /api/ask 는 클라이언트가 보낸 스냅샷을 신뢰한다(라우트에 이미
  authoritative-read TODO 주석 있음). 툴이 쓰는 sessionId 도 클라이언트 제공값 — events 는
  공개 읽기라 저위험이나, 클라이언트가 임의 sessionId 를 넣을 수 있음을 인지한다
- **읽기 볼륨·캐시**: 같은 레이스를 묻는 동시 사용자들은 동일 events 를 읽는다 → 핫쿼리 캐시
  여지. 인기 레이스 × 툴 라운드 × 사용자로 읽기가 증폭되지 않게 봉투를 잡는다(수용 기준 §참조)

### 툴 세트 (관찰로 2개로 확정)

| 툴 | 출처 | 판정 |
|---|---|---|
| `queryDriverEvents(driver, type?, lapRange?)` | Firestore events | **핵심.** pit_stop·investigation·penalty·strategy_note·track_hazard. A·B 의 "예/부분" 대부분이 이 툴 하나로 해결. **investigation 을 반드시 포함**(인과의 실질 재료). lapRange 는 lap 이 있는 타입에만(overtake·fastest_lap 제외) |
| `lookupF1Knowledge(topic)` | 큐레이션 지식 파일 | **필수.** C 유형 100% 가 여기에만 의존 |
| ~~`queryStints`~~ | ~~contextSummary.stints~~ | **드롭 (R3 실측).** stints 22명 전원이 **이미 스냅샷에 실려 프롬프트에 도달**한다 → 툴로 다시 조회해도 새 데이터 0·순수 왕복 지연. 스틴트 질문은 이미-실린 데이터로 답됨 |
| `queryPace(driver, lapRange?)` | 저장 결정 종속 | **조건부.** §A 결정대로 스틴트 페이스 요약을 스냅샷에 실으면 이 툴 없이 프롬프트로 대부분 커버. 세밀 구간만 온디맨드 1요청 |

**프로토타입 최소 세트: `queryDriverEvents` + `lookupF1Knowledge` 2개.** 툴 많으면 모델이 헤맨다.

### 라우터 (지연 관리 — "항상 열기"로 확정)

세컨드 스크린은 지연이 중요하다. 관찰 실측: 단순 질문 5/5("지금 누가 선두"·"3위 누구"·"몇
랩째"·"선두 갭"·"날씨")가 **스냅샷+narrative 만으로 답 가능, 툴 0회**.

- **채택: 항상 툴을 열어두되, 프롬프트로 "이미 실린 데이터로 답할 수 있으면 툴을 쓰지 말라"를
  강제**한다. "먼저 툴 필요?를 묻는" 방식은 모든 질문에 강제 1콜 세금을 매기지만, 이 방식은
  모델이 안 쓰면 단발과 동일(1콜)이라 더 우월
- **진짜 리스크는 오발동**: 단순 질문에서 모델이 불필요하게 툴을 호출해 1콜이 2콜 왕복이 되는
  것. **오발동률**(단순 질문 셋에서 툴 헛호출 비율)을 프로토타입에서 측정·억제한다(수용 기준 §참조)

## 순서 (1단계처럼 관찰 게이트 먼저)

1. **관찰 — 완료.** 깊은 질문 29개 매핑, 답 가능률(A 63/88%·B 29/71%·C 0%·D 부분), 툴 2개 확정,
   랩타임 저장 결정(스틴트 요약+온디맨드), 큐레이션 최소 범위 산출
2. **프로토타입**: Gemini(라이브 provider) **하나에만** 툴 루프 + 서버 Firestore +
   `queryDriverEvents`(+지식) 로 "VER 몇 랩 피트?"·"누가 언더컷?"이 40건 창 무관하게 답되는지
   확인. **먼저 벨기에 픽스처에 overtakes 엔드포인트를 보강**한다 — 현 픽스처엔 overtake 데이터가
   전무해(overtakes 0건, 관찰 발견) 추월·언더컷 답변력을 과소평가한다. 3 provider 는 그 다음
3. 확정된 툴·라우터를 3 provider 로 일반화, MockLlmProvider 대응, 큐레이션 지식 파일,
   스틴트 페이스 요약 저장 확대

## 범위 밖

- 랩타임 세분 텔레메트리(섹터별 풀 이력) — 저장 결정에서 과하면 제외
- **전체 결과 예측·반사실**("누가 이길까", "피트했어야 했나") — 결정론 재료 없음, 명시적 범위 밖.
  인접 페어 클로징만 기존 overtake_forecast(docs/23)로 연결
- 과거 경기 교차 질문("지난 레이스 대비") — 아카이브 별건
- 큐레이션 지식의 자동 갱신 — 수동 관리(검증된 사실만)
- 툴 실행을 클라이언트가 하는 변형 — 서버측으로 확정(사용자 선택)

## 수용 기준

1. 관찰(완료): 깊은 질문 유형별 데이터 출처 매핑표 + 랩타임 저장 결정 + 인과 답 가능률
2. Gemini 프로토타입이 "VER 몇 랩 피트?"·"누가 언더컷?"에 전체 이력으로 답한다(40건 창 무관)
3. 서버 Firestore 접근이 읽기 전용·서비스계정 서버 환경변수·클라이언트 미노출
4. 툴 정의·실행은 공용, provider별은 wire 어댑터만 (세 벌 안 만든다)
5. **인과 두 경로 모두 테스트**: (a) 데이터에 없으면 "기록에 없다"고 답하고 추측 안 함,
   (b) 인접 사건은 "~때문일 수 있다"로 상관 제시하되 단정("~때문이다") 안 함 (R6)
6. 규정·상식은 큐레이션 지식 파일로 답하고 지어내지 않음. **모든 지식 항목에 출처 주석 존재를
   테스트로 강제**(R5 — C 는 결정론이 안 지켜주므로)
7. 간단한 질문은 툴 없이 답(지연 회귀 없음). **단순 질문 셋 오발동률 측정·억제**(툴 헛호출 0 목표, R7)
8. MockLlmProvider 가 인터페이스 만족, FallbackLlmProvider 통과
9. 도메인 단위 테스트: 각 툴의 결정론 조회(드라이버·타입·랩 범위 필터), 지식 조회.
   **queryDriverEvents 가 investigation 을 포함**하고, lapRange 가 lap 없는 타입(overtake·
   fastest_lap)에 안전
10. 벨기에 픽스처로 깊은 질문 회귀(VER 피트 랩이 실제 pit 이벤트와 일치). **overtakes 보강 후**
11. **툴 루프 종료 상한**: 최대 라운드 캡에서 강제 최종 답변, 무한 반복 없음 (D1, 테스트)
12. **깊은 경로 예산**: 질문당 툴 라운드·Firestore 읽기·토큰 상한을 정하고 초과 안 함 (D2)
13. **각 질문 독립**: 크로스턴 툴 상태 없음, 후속 질문은 툴 재실행 (D3)
14. **복합 인덱스 배포**가 코드 머지와 별도로 반영됨 (driverNumber+type+lapNumber)
