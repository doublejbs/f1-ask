# 23. 추월 예측 — 랩타임으로 "몇 랩 후"를 계산한다

## 배경

제품 축 전환(docs/19)의 연장. 방송(F1 TV Track Pulse)은 이미 두 가지를 잘한다:

1. **현재** 따라잡는 속도 표시 — "BOT Catching OCO, Catch Rate -0.923"
2. **1초 이내** 배틀 실황 — TV 해설이 실시간으로 중계한다

우리가 할 일은 그 앞이다. **"이 페어는 N랩 후에 배틀 범위(1초 내)에 들어간다"** 는
예측은 방송이 안 해주고, 시청자가 암산하기도 어렵다. 세컨드 스크린의 역할 분담:

- 1초 이내 → TV 영역. 우리는 손대지 않는다 (기존 battle 표시로 충분)
- **1.5초~수십 초 → 우리 영역.** 랩타임 추세로 진입 랩을 예측한다

## 원칙: 예측은 산수다

docs/02 §3.1 결정론적 코어 그대로. 추월 예측의 본체는 나눗셈이다:

```
잡는 속도(초/랩) = 최근 N랩 동안 (앞차 랩타임 − 뒷차 랩타임) 평균
남은 랩 수      = (현재 interval − 배틀 임계 1.0초) ÷ 잡는 속도
```

LLM 은 여기에 없다. LLM 의 자리는 **"왜 잡히는가"** — 타이어 나이 22랩 vs 7랩,
하드 vs 미디엄 같은 이유 설명이며, 이것이 유료 AI 해설의 가치다 (유료 구조 자체는
별도 스펙). 숫자는 결정론이라 환각이 없고, 이유는 LLM 이라 해설이 된다.

## 설계

### 계산 위치: 워커

contextSummary(docs/22 B)와 동일한 패턴. 예측에 필요한 랩타임 이력은 원본
`laps` 에만 있고 Firestore 스냅샷엔 `lastLapSeconds` 한 개뿐이다. 원본을 쥐는
유일한 지점이 워커이므로 워커가 계산해 스냅샷에 싣는다. 클라이언트는 계산하지
않는다 (아키텍처 원칙).

구현 위치: `buildOpenF1LiveFrame` 내부에서 `buildOvertakeForecasts(data, nowMs)`
호출 — 워커·리플레이·아카이브가 자동으로 공유된다.

### 대상 페어: 순위 인접

MVP 는 **순위 인접 페어(Pn ↔ Pn+1)만** 본다. interval 데이터가 직접 있고, 추월이
실제로 일어나는 단위다. 제외:

- 어느 한쪽이 랩다운(+1 LAP 이상)·리타이어·interval null
- 세션이 레이스 진행 상태가 아닐 때 (SC·VSC 포함 — 전 차량이 인위적으로 밀착돼
  랩타임·간격 모두 예측 재료가 못 된다. WatchNowDetector 의 `isRacingStatus` /
  `isSafetyCarStatus` 판정 재사용)

F1 TV 의 "BOT Catching OCO (55초 차)" 같은 비인접 롱레인지는 범위 밖.

### 잡는 속도: 최근 N랩, 오염 랩 제거

- 양쪽 모두 `lap_duration` 이 있는 **공통 최근 N=3랩** 의 delta 평균
- **피트 오염 랩 제외**: `pits` 의 `lap_number`(인랩)와 그 +1(아웃랩)은 양쪽 모두
  계산에서 뺀다. 우리 `OpenF1Lap` 엔 `is_pit_out_lap` 이 없지만 pits 는 이미
  조회하고 있으므로 추가 요청이 없다
- **이상치 제외**: 드라이버 본인의 최근 유효 랩 중앙값 대비 +5% 초과 랩은 뺀다
  (SC 랩·트래픽 랩·스핀이 걸러진다)
- 유효 랩이 N개 미만이면 그 페어는 예측하지 않는다 (억지로 만들지 않는다)

### 발화 조건 (전부 만족해야 예측)

| 조건 | 값 | 이유 |
|---|---|---|
| interval | > 1.5초 | 그 아래는 TV 영역, 기존 battle 표시가 이미 담당 |
| 잡는 속도 | ≥ 0.15초/랩 | 노이즈 수준의 미세 접근을 예측이라 부르지 않는다 |
| 예측 랩 수 | ≤ 10랩 | 10랩 밖 예측은 타이어 열화·피트로 무의미 |
| 예측 랩 수 | ≤ 남은 랩 | 체커기 전에 못 잡으면 예측이 아니다 |

임계값은 `OvertakeForecastConfig` 로 분리한다 (WatchNowDetectorConfig 패턴).
**값은 벨기에 GP 픽스처 실측으로 튜닝한다** — 먼저 측정하고, 숫자에 맞추지 않는다
(docs/20 교훈).

### 스냅샷 계약

`LiveRaceSnapshot` 에 optional 필드 추가 (mock·replay·옛 스냅샷 안전):

```ts
overtakeForecasts?: OvertakeForecast[];

type OvertakeForecast = {
  chaserNumber: number;      // 뒷차 (따라잡는 쪽)
  targetNumber: number;      // 앞차
  intervalSeconds: number;   // 예측 시점 간격
  closingRateSecondsPerLap: number; // 0.01 단위 반올림
  predictedLapsToBattle: number;    // 정수 반올림 (올림 — 낙관하지 않는다)
  predictedLap: number;      // currentLap + predictedLapsToBattle
};
```

반올림해 저장하는 이유: 폴링마다 소수 끝자리가 흔들려 PublishDecision 지문을
불필요하게 바꾸는 것을 막는다. 예측 입력(interval·랩타임)은 어차피 스냅샷 본체라
본체 변화에 동반 갱신되므로 contextSummary 처럼 지문에서 제외하지 않는다 —
**리플레이로 스냅샷 쓰기 횟수가 늘지 않음을 실측 확인한다.**

AI 질문 컨텍스트는 스냅샷을 그대로 받으므로 예측이 자동으로 들어간다 (docs/22 와
같은 경로). "누가 추월할 것 같아?" 에 결정론 데이터로 답할 수 있게 된다.

### 이벤트: `overtake_forecast` (엣지 트리거)

예측이 **처음 성립하는 순간** 1회만 이벤트를 낸다. 매 폴링 재발화하면 overtake
214건 소음(docs/22 배경)의 재판이 된다.

- 새 `RaceEventType.OvertakeForecast = "overtake_forecast"`, scope 는 Driver(chaser),
  priority 는 medium
- params: chaser·target·predictedLap·closingRate (스냅샷 필드와 동일 값)
- 재무장 조건은 두 갈래다 (`OvertakeForecastTracker` 가 프레임 간 상태를 든다 —
  WatchNowDetector 패턴, 순수 함수 + 상태 클래스 분리):
  - **하드 해체 (즉시 재무장)**: 둘 중 하나 피트(pitStopCount 증가)·리타이어, 또는
    두 드라이버가 더 이상 순위 인접이 아님(스왑·제3자 개입). 진짜 에피소드 경계라
    디바운스 없이 즉시 활성 상태를 푼다
  - **소프트 소멸 (랩 디바운스)**: 페어는 여전히 순위 인접인데 forecasts 목록에서만
    빠짐(발화 조건 잡음 이탈). interval·잡는 속도가 임계값 주변을 폴링 잡음으로 오가면
    예측이 한두 프레임 빠졌다 돌아오는데(flicker), 즉시 재무장하면 매번 재발화한다.
    그래서 **최소 한 랩 온전히 부재**(`absentSinceLap + 2`, 트래커 옵션 기본 상수)해야
    진짜 소멸로 보고 재무장한다. currentLap 이 null 인 프레임에선 판정을 보류하고 상태
    유지
  - 그리고 **성립 순간 1회를 랩 단위로 강제**한다: 한 페어는 같은 랩에 두 번 발화하지
    않는다. 하드 해체는 활성을 즉시 풀지만(재무장은 즉시), 피트 아웃랩 재등장·순위
    데이터 지터가 같은 랩 재발화를 만들 수 있어, 재발화는 랩이 넘어간 뒤에만 허용한다
- **AI 해설 대상**: `CommentaryEventAllowlist` 에 추가. 해설 프롬프트에 양쪽의
  compound·tireAgeLaps 가 이미 들어가므로 "왜 잡히는가" 를 지어내지 않고 설명할
  수 있다. `EVENT_TYPE_GUIDANCE` 에 overtake_forecast 전용 지침 추가 — "예측 랩을
  단정하지 말고 추세로 말할 것, 타이어 차이를 근거로 들 것"
- 질문 컨텍스트: `QuestionEventAllowlist` 는 CommentaryEventAllowlist 를 상속하므로
  자동 포함

### UI: 지금 볼 것 레인

예측은 정의상 "다음에 볼 것"이다. 별도 화면을 만들지 않고 기존 watch-now 칸
구조(선두권/필드/내 드라이버)에 신호 타입으로 통합한다:

- `WatchNowSignalType` 에 forecast 추가, 문구는 "NOR, 3랩 후 BOR 1초 내 진입 예상"
- 칸 배치는 기존 규칙 그대로 (chaser 위치 기준). 즐겨찾기 드라이버가 chaser 또는
  target 이면 내 드라이버 칸
- 이벤트 피드에는 overtake_forecast 이벤트가 기존 경로로 흐른다 (i18n 번역 추가)

## 검증

1. 도메인 단위 테스트: 잡는 속도 계산(피트 오염 랩 제외·이상치 제외·유효 랩 부족),
   발화 조건 경계, 트래커 엣지 트리거·재무장
2. **벨기에 GP 픽스처 실측**: 어떤 페어가 몇 건 예측되는지 먼저 관찰 → 임계값
   확정 → 회귀 테스트로 고정. 스크린샷의 실제 배틀(HUL→LAW 0.595 등)이 그 전
   랩들에서 예측으로 잡혔는지 교차 확인

   실측 결과(2026-07, 15초 스텝 339프레임, `OvertakeForecastMeasurement.test.ts`):
   flicker 수정 전 46페어에 **79발화**(활성 대비 1.72배, 같은 랩 중복 6건, 최다 SAI→ALB
   7회)에서 랩 디바운스 + 랩당 1회 상한으로 **57발화**(고유 페어 46개 유지, SAI→ALB 3회,
   같은 랩 중복 0)로 줄었다. 적중 46%·조기 32%(예측보다 빨리 배틀 진입 — 임계값이
   보수적)·지연 7%·미적중 16%. **HUL→LAW 는 실제 배틀(L35~39) 이전인 L34 에 사전
   포착**돼 예측 기능의 존재 이유를 실증한다. 피트 오염 랩 필터는 잡는 속도 왜곡을 막는
   핵심으로, 수정 전 트래커에선 필터 OFF 가 +20건의 허위 발화를 냈다(수정 후엔 랩
   디바운스가 그 flicker 를 흡수해 차이가 −3건으로 줄어든다). 임계값(1.5s / 0.15 / 10랩)은
   이 실측으로 확정.
3. 리플레이: 스냅샷 쓰기 횟수 증가 없음 + 이벤트 소음 없음(페어당 재발화 확인)
4. 실전 배포 전 프로덕션 리플레이로 end-to-end (docs/20 하네스 재사용)

## 범위 밖

- 비인접 페어 롱레인지 예측 (F1 TV 의 BIG CATCH 류)
- 무료/유료 entitlement 구조 — 별도 스펙 (다음)
- 팀라디오 선수별 보기·번역 — 별도 스펙
- 외부 뉴스 피드 — 별도 스펙
- 피트 전략 시뮬레이션 (언더컷 예측 등 — watch-now 언더컷 감지는 기존 것 유지)

## 수용 기준

1. 워커가 순위 인접 페어의 배틀 진입 랩을 결정론적으로 계산해 스냅샷에 싣는다
2. 피트 오염 랩·이상치가 잡는 속도 계산에서 제외된다
3. 발화 조건 4개(간격·속도·예측 랩 상한·남은 랩)를 전부 만족할 때만 예측한다
4. `overtake_forecast` 이벤트는 페어당 성립 순간 1회만 나오고, 해체 후 재성립 시에만
   재발화한다
5. AI 해설이 overtake_forecast 를 다루며 타이어 근거를 포함한다
6. 스냅샷 필드는 optional 이라 mock·옛 스냅샷에서 안전하다
7. 벨기에 GP 픽스처 회귀 테스트가 예측 페어·건수를 고정한다
8. 리플레이에서 스냅샷 쓰기 횟수가 예측 추가 전과 같다
